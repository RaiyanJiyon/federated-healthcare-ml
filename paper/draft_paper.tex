\begin{abstract}
Traditional predictive AI in healthcare usually hits a wall pretty early: patient data is stuck in silos. Because of strict privacy rules like the Health Insurance Portability and Accountability Act (HIPAA) and the General Data Protection Regulation (GDPR), clinical facilities can't just pool their data into one big central pot, which makes training a strong model nearly impossible. Federated Learning (FL) looks like a great fix since it keeps training decentralized, but we rarely see it tested against the messy, practical constraints of a real clinic.

In this paper, we built a Federated Learning framework specifically for the clinical world, using an imbalanced diabetes dataset to keep things realistic. We started by mimicking the demographic gaps you'd actually see between different hospitals, distributing data across network proxies using a Non-IID Dirichlet partition (alpha = 0.5). To keep the network locked down, we pulled in Differentially Private Stochastic Gradient Descent (DP-SGD) and added a mathematical shield against bad-actor injections using robust Byzantine aggregation (specifically Krum and Median).

The real breakthrough, though, was shifting our focus. Instead of chasing raw accuracy, we prioritized clinical safety. By tweaking custom thresholds and refining how the model interacts with data, we managed to push critical diagnostic Recall from a dangerous 70\% baseline all the way up to 87.04\%. It’s proof that privacy-first AI doesn't have to compromise on the metrics that actually keep patients alive across isolated healthcare silos.
\end{abstract}

\section{Introduction}
Today, healthcare systems store huge amounts of patient data. This creates a big chance for us to use machine learning to help doctors diagnose diseases. But there is a problem. In a traditional machine learning approach, you have to send all the data to a cloud platform so the models can be trained. This is not safe and it violates data security laws like HIPAA or GDPR. Because of these security issues, authorities can't share their data, so patient records stay stuck in isolated databases across different hospitals and clinics \cite{price2019privacy}.
Federated Learning (FL) was created to solve this. It lets us train models across different healthcare sites without moving the private data or breaking privacy rules. In an FL setup, each hospital trains the model on its own local data. Then, they only send the updated model weights to a central server, not the actual patient files. Even though this sounds good, most current FL studies use perfect, evenly distributed public datasets. Real clinical data is messy and varies a lot from one hospital to another.

In this research, we built and tested an FL system using actual clinical diabetes data. We wanted to see if this works under the strict rules hospitals have to follow. Our work focuses on three main things:
\begin{itemize}
\item We used a Dirichlet-based distribution to mimic how data is actually split between different hospitals.
\item We used Krum and Median aggregation methods. These help protect the global model if one of the sites has bad or "poisoned" data.
\item We focused on "Recall" rather than just overall accuracy. We reached an 87.04\% detection rate because catching every case of diabetes is what actually keeps patients safe.
\end{itemize}

The remainder of this paper is organized as follows: Section II reviews related literature and existing theoretical gaps. Section III details our methodology, including data distribution, robust aggregation, and privacy constraints. Section IV evaluates our empirical results and scalability benchmarks. Finally, Section V concludes the research narrative and discusses future work.

\section{Related Work}

Researchers have been looking into Federated Learning for healthcare because it helps keep data in the hands of its owners \cite{kairouz2021advances}. Most previous tests use balanced, simple datasets and standard math formulas like FedAvg to combine data \cite{mcmahan2017communication}. However, using these same algorithms in real clinics is much harder because medical data is often missing pieces or spread out unevenly.

Some papers have tried to fix medical data silos by changing how much weight they give to certain features or using decentralized methods. But these don't really consider what happens if the network is unstable or if someone tries to upload bad data on purpose \cite{li2020federated_challenges}. When the data is split unevenly across different sites—which is usually the case in the real world—standard averaging methods start to break down and become unstable \cite{zhao2018federated, li2020federated}.

More recently, people have started using tougher rules like Krum \cite{blanchard2017machine} and Median \cite{yin2018byzantine} to stop "poisoned" data from ruining the model. These work well in theory, but we still don't know exactly how much they hurt "Recall"—which is how often the model correctly catches a disease. If these security tools make the model miss sick patients, that's a big safety issue. This paper aims to fill that gap by testing how these security measures affect patient safety in a real diagnostic pipeline.

\section{Methodology}
We built a distributed setup that acts like several independent hospitals to see how Federated Learning (FL) works when things get complicated in the real world. This section explains how we set up the system, how we split up the uneven data, and the tools we used to defend against bad data.

\subsection{System Setup}
Our system uses a central server that talks to different clients using the Flower framework. We set up 5 local proxies (acting like hospitals) that go through 10 rounds of training. Because the data isn't even, each hospital trains its own Logistic Regression model using the L-BFGS algorithm. We set the learning rate to 0.01 and let the local training run up to 2000 times to make sure it’s steady before it sends anything back. The most important part is that we never share the actual patient data. We only send the mathematical model weights back to the main server to be combined.

\subsection{Splitting the Patient Data (Non-IID)}
A big problem in a lot of AI research is the assumption that data is perfectly shared and identical everywhere. In actual practice, different hospitals see very different types of patients. To make our test realistic, we split the data unevenly.

We used a Dirichlet distribution with a setting of 0.5 to split the patient records. This means some "hospitals" in our test end up with way more diabetic patients than others. This forces the local models to deal with biased information, which really tests if the main global model can still learn correctly from such a messy mix of data.

\subsection{Combining the Models}
Normally, the central server simply combines the hospitals’ updates by averaging them through a protocol called FedAvg. But when the data is split unevenly like ours, just averaging everything can make the global model unstable.

To fix this and protect the system from "poisoned" or bad updates, we used stronger rules. We compared the standard average to the Median (which ignores extreme outliers) and the Krum method. Krum works by picking the update that is closest to its neighbors, which mathematically blocks out any "toxic" data that a hacked or broken hospital might try to send.

\subsection{Keeping Data Private}
To follow strict healthcare privacy rules, we used Differential Privacy (DP-SGD). Before any hospital sends its updates to the server, we add a bit of mathematical "noise" (Gaussian noise) to the data. We set the privacy budget to 1.0 and a failure chance of 0.01. This makes it extremely hard to trace the model back to the original patients. And even with this additional protection, the model’s accuracy dropped by only 2.1\%, while it still correctly identified 87\% of diabetes cases. This shows that we can protect privacy without making the medical results much worse.

Given the deployment context of diagnostics, our evaluation framework decisively diverges from conventional ML evaluations by deprioritizing gross accuracy. In clinical environments, a false negative (missed diagnosis) is significantly more catastrophic than a false positive (which simply prompts secondary testing).

\subsection{Clinical Baseline vs. Federated Optimization}
We established a centralized training baseline for comparison. The centralized model initially achieved an accuracy of 73.38\% but recorded a dangerous Recall (Sensitivity) metric of only 70.37\%, resulting in a 30\% failure rate (16 missed diabetic patients out of a sample 54).

By optimizing the binary classification decision threshold boundary--lowering it from 0.5 to 0.30--and generating 11 complex interaction terms (e.g., Glucose \( \times \) BMI) natively on the clients, our federated application achieved a fundamentally superior clinical profile. Furthermore, performance trajectory monitoring confirmed that global convergence strictly stabilized at 10 communication rounds, with subsequent empirical tests indicating no marginal Recall improvement when redundantly scaling past 15 consecutive rounds. As detailed in Table \ref{tab:performance}, the FL implementation drove the Recall to 87.04\%, actively reducing the volume of missed diagnoses from 16 patients down to just 7. This establishes the federated network as operationally viable for preliminary healthcare deployments.

\subsection{Privacy and Scalability Efficiency Trade-Offs}
While DP-SGD ensures gradient security, and Non-IID Dirichlet distribution guarantees demographic realism, they induce an expected performance taxation. Our scalability testing evaluated system convergence across configurations scaling from 5 incrementally up to 20 clients concurrently (see Fig. \ref{fig:comprehensive}). The system demonstrates asymptotic convergence efficiency, wherein communication latency scales linearly with $N$ clients without degrading the clinical safety thresholds required for the 87\% recall boundary up to 15 clients (see Fig. \ref{fig:scaling}). The marginal degradation in Recall observed when scaling from 15 to 20 clients is mathematically attributed to extreme feature dilution; because the fixed dataset size is partitioned excessively thinly, individual proxy clients fail to capture sufficient positive-class variance to compute robust, stabilizing local gradients before central aggregation. To further validate realistic operational constraints, we systematically simulated unannounced client dropouts mid-round; the global architecture maintained zero structural degradation, requiring only standard fallback omission to complete the aggregation round flawlessly, proving absolute resilience against unpredictable hospital network latency (see Fig. \ref{fig:bottleneck}). We observed that implementing Byzantine robust aggregations such as Krum requires roughly a 15-20\% increase in required communication rounds to reach standard stabilization, presenting a quantifiable trade-off between absolute system safety and sheer bandwidth optimization.

\section{Conclusion}
This paper demonstrates that Federated Learning is not merely a theoretical privacy mechanism, but a clinically viable methodology for decentralized predictive operations under realistic constraints. By evaluating the system against severe Non-IID data distributions designed to simulate real-world hospital disparity, our framework actively mirrors clinical reality. Furthermore, the integration of Differential Privacy and Byzantine-resilient Krum aggregation guarantees the safe operation of distributed machine intelligence within heavily audited regulatory infrastructures. 

Crucially, our study highlights that gross accuracy is an insufficient metric for medical ML. By applying custom threshold optimization and localized feature engineering, we traded marginal precision boundaries in order to increase critical diagnostic Recall to 87.04\%, minimizing dangerous false-negative patient outcomes. Future research must expand computational evaluations of robust aggregators across non-horizontal partitioning spaces in order to fully scale federated networks into active diagnostic environments.
